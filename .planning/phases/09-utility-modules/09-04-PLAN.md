  <action>
1. **CrashLogsPanel.tsx** — Crash log viewer:
   - Device selector at top (same pattern as SettingsPanel — dropdown with booted devices)
   - App name filter input and date range filter (since date picker or text input for ISO date)
   - "Refresh" button to reload logs
   - On device select (and on filter change), fetch `GET /api/modules/crash-logs/list/<deviceId>?app=<filter>&since=<date>`
   - Display crash logs in a scrollable list. Each entry shows:
     - Timestamp (formatted readable)
     - Process name
     - Preview text (truncated to 2 lines)
   - Click on an entry to view full content: fetch `GET /api/modules/crash-logs/view/<deviceId>/<logId>`
   - Show full content in a glass-panel code block (pre/code element) with monospace font, scrollable
   - "Back to list" link to return from detail view
   - Empty state: "No crash logs found" when list is empty
   - Glass-panel styling consistent with other panels
   - Call `registerPanel("crash-logs", CrashLogsPanel)` at module scope (inline pattern, same as SettingsPanel)

2. **MediaPanel.tsx** — Media injection with drag-and-drop:
   - Device selector at top (same pattern)
   - Large drag-and-drop zone in the center:
     - Dashed border, glass-panel style
     - Text: "Drop photos or videos here to add to device"
     - Shows accepted file types: .jpg, .png, .gif, .mp4, .mov, .heic
     - Also has a "Browse files" button that opens a hidden file input
   - Implement drag-and-drop handlers (onDragOver, onDragEnter, onDragLeave, onDrop):
     - On drop: extract files from DataTransfer
     - For each file, create FormData with the file and POST to `/api/modules/media/add/<deviceId>`
     - Show uploading state (spinner or progress text)
     - Toast success: "Added <filename> to <device.name>"
     - Toast error on failure
   - Visual feedback: highlight drop zone border on drag-over (change border color to accent-blue)
   - Follow the drag-and-drop pattern from `packages/dashboard/src/panels/apps/InstallDropZone.tsx` for reference
   - Call `registerPanel("media", MediaPanel)` at module scope

3. **ClipboardPanel.tsx** — Clipboard bridge:
   - Device selector at top (same pattern)
   - Two sections in a single glass-panel:
   
   **Read Clipboard section:**
   - "Read Clipboard" button
   - On click, fetch `GET /api/modules/clipboard/get/<deviceId>`
   - If 400 (unsupported), show toast error "Clipboard read not supported for this platform"
   - Display clipboard content in a readonly textarea (glass-panel styled, monospace)
   - "Copy to Host" button next to the textarea that uses `navigator.clipboard.writeText()` to copy to host clipboard, with toast "Copied to host clipboard"
   
   **Write Clipboard section:**
   - Textarea for entering text to write to device clipboard
   - "Write to Device" button
   - On click, POST to `/api/modules/clipboard/set/<deviceId>` with `{ text }`
   - Toast success: "Clipboard set on <device.name>"
   - "Paste from Host" button that reads host clipboard via `navigator.clipboard.readText()` and populates the textarea, then auto-sends to device
   
   - Call `registerPanel("clipboard", ClipboardPanel)` at module scope

All panels follow established conventions:
- Use `fetch()` for API calls
- Use `sonner` toast for feedback
- Use oklch glass-panel styling with Tailwind classes
- Device selector dropdown with same styling as SettingsPanel
- Consistent error handling (try/catch, toast.error)
