Instead, this version should NOT take a wsRef prop. Instead, accept a `send` prop of type `(msg: Record<string, unknown>) => void` that the parent (LocationPanel) will provide. The parent will wrap simvyn's `useWs().send()` to convert flat messages to envelope format. The PlaybackControls component itself should send flat-style messages like `{type: "start-playback", waypoints, speedMs, multiplier, loop, deviceIds}` — the parent's `send` wrapper will handle the WS envelope wrapping.

Wait — actually, simpler approach: PlaybackControls should accept a `sendLocation` prop `(type: string, payload: Record<string, unknown>) => void` that sends to the location channel. The component calls `sendLocation("start-playback", { waypoints, speedMs, ... })` etc. The parent provides: `(type, payload) => send({ channel: "location", type, payload })`. This avoids leaking WS internals into the component.

Full features: play/pause/stop buttons, speed preset dropdown (Walking/Cycling/Driving/Train/Plane), custom speed number input, km/h vs m/s unit toggle, 1x/2x/5x/10x multiplier buttons, loop toggle, progress bar with percentage. On speed change during playback, sends `update-playback-speed` message. Reads device selection from simvyn's global `useDeviceStore` — get `selectedId` from `useDeviceStore` to pass as `deviceId` in playback messages. NOTE: sim-location uses multi-device selection (Set of IDs, broadcastMode). Simvyn's dashboard uses single device selection via TopBar DeviceSelector. For playback start, get `selectedId` from simvyn's global device store and pass as `deviceId` (string, not array).
