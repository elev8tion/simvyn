**ios-fs.ts** — iOS simulator file operations via direct Node.js fs access:
- `iosListDir(containerPath: string, relativePath: string)`: Use `readdir(fullPath, { withFileTypes: true })` + `stat()` to return entries with name, path (relative), isDirectory, size (bytes), modified (ISO string). The containerPath is the host filesystem path from `dataContainer` on AppInfo (e.g., `/Users/.../Containers/Data/Application/UUID`).
- `iosReadFile(containerPath: string, relativePath: string)`: Return `Buffer` via `readFile(join(containerPath, relativePath))`.
- `iosWriteFile(containerPath: string, relativePath: string, data: Buffer)`: Write via `writeFile(join(containerPath, relativePath), data)`.
- `iosGetContainerPath(deviceId: string, bundleId: string)`: Use `execFileAsync("xcrun", ["simctl", "get_app_container", deviceId, bundleId, "data"])` and return the trimmed stdout. Always resolve fresh — never cache (container UUIDs change on reinstall).

**android-fs.ts** — Android file operations via adb shell run-as:
- `androidListDir(deviceId: string, packageName: string, remotePath: string)`: Execute `adb -s <deviceId> shell run-as <packageName> ls -la <remotePath>`. Parse the output defensively — each line has permissions, links, owner, group, size, date, name. Skip the "total" header line. Return entries with name, path (relative), isDirectory (starts with 'd'), size, modified.
- `androidPullFile(deviceId: string, packageName: string, remotePath: string)`: Copy via run-as to `/data/local/tmp/simvyn_transfer`, then `adb pull` to a host temp dir, read into Buffer, clean up both temp files. Use `mkdtemp` for host temp.
- `androidPushFile(deviceId: string, packageName: string, localPath: string, remotePath: string)`: `adb push` local file to `/data/local/tmp/simvyn_transfer`, then `run-as cp` to final location, clean up staging file.
- `androidGetContainerPath(deviceId: string, packageName: string)`: Execute `adb -s <deviceId> shell run-as <packageName> pwd` to verify debug access (throws if not debuggable), return `/data/data/<packageName>` as the container path.
- Guard against `avd:` prefix device IDs in all functions — throw "Device must be booted" (consistent with existing android adapter pattern).
- Catch "not debuggable" errors from run-as and throw a clear error: "Package <name> is not debuggable. Only debug builds can be inspected."

Use `execFile` (never `exec`) via `promisify` for all shell commands. Use `mkdtemp` + `rm` with finally blocks for temp dir cleanup.

**routes.ts** — Fastify REST routes for file operations (follow app-management/routes.ts pattern):

Register `@fastify/multipart` scoped to this plugin (limits: { fileSize: 100_000_000 } — 100MB for file uploads).

Helper function `resolveContainer(deviceId, bundleId, fastify)`: Look up device from `fastify.deviceManager.devices`, validate booted state, get adapter via `getAdapter(device.platform)`, call `adapter.getAppInfo(deviceId, bundleId)` to get dataContainer. For iOS, the dataContainer is a host path used directly. For Android, it's a device path used with adb run-as.

Routes (all prefixed under /api/modules/fs by the module system):

1. `GET /ls/:deviceId/:bundleId` with query param `path` (default "."): List directory contents. Dispatch to iosListDir or androidListDir based on platform. Return `{ entries: FileEntry[] }` where FileEntry is `{ name, path, isDirectory, size, modified }`.

2. `GET /pull/:deviceId/:bundleId` with query param `path` (required): Download a file. For iOS: readFile and send as buffer with appropriate content-disposition header (attachment; filename=<basename>). For Android: pull to temp, send buffer, clean up. Set Content-Type to application/octet-stream.

3. `POST /push/:deviceId/:bundleId` with query param `path` (required): Upload a file via multipart. Get file from `req.file()`, save to temp dir via pipeline, then write to device (iOS: direct writeFile to container; Android: push via adb). Return `{ success: true }`.

4. `GET /read/:deviceId/:bundleId` with query param `path` (required): Read text file content. Same as pull but return `{ content: string, path: string }` — convert Buffer to UTF-8 string. Only serve for known text extensions: .txt, .json, .xml, .plist, .sql, .csv, .html, .css, .js, .ts, .md, .log, .cfg, .ini, .yaml, .yml. For .plist files on iOS, run through `plutil -convert xml1 -o -` first to get human-readable XML. Return 400 for binary files.

5. `POST /write/:deviceId/:bundleId` with body `{ path: string, content: string }`: Write text content back to file. Convert string to Buffer, write via platform adapter. For iOS plist files that were converted to XML for reading, convert back to binary plist via `plutil -convert binary1 -o <path> -` before writing. Return `{ success: true }`.
