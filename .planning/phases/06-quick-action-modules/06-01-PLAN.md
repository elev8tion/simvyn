  <action>
    Add optional methods to the PlatformAdapter interface in device.ts:
    - `screenshot?(deviceId: string, outputPath: string): Promise<void>` — capture screenshot to file
    - `startRecording?(deviceId: string, outputPath: string): Promise<ChildProcess>` — start recording, return child process
    - `stopRecording?(process: ChildProcess): Promise<void>` — stop recording gracefully

    Import ChildProcess type: `import type { ChildProcess } from "node:child_process"` in device.ts.

    iOS adapter (ios.ts) — implement:
    - `screenshot`: `execFileAsync("xcrun", ["simctl", "io", deviceId, "screenshot", outputPath])`
    - `startRecording`: `spawn("xcrun", ["simctl", "io", deviceId, "recordVideo", outputPath])` — returns the child process. simctl records until the process is killed.
    - `stopRecording`: send SIGINT to the child process (`process.kill("SIGINT")`), then wait for exit via a promise wrapping the `close` event. SIGINT causes simctl to finalize the video file gracefully.

    Android adapter (android.ts) — implement:
    - `screenshot`: Two-step: (1) `execFileAsync("adb", ["-s", deviceId, "shell", "screencap", "-p", "/sdcard/simvyn_screenshot.png"])`, (2) `execFileAsync("adb", ["-s", deviceId, "pull", "/sdcard/simvyn_screenshot.png", outputPath])`, (3) cleanup: `execFileAsync("adb", ["-s", deviceId, "shell", "rm", "/sdcard/simvyn_screenshot.png"])`. Guard against `avd:` prefix IDs (same pattern as app-management).
    - `startRecording`: `spawn("adb", ["-s", deviceId, "shell", "screenrecord", "/sdcard/simvyn_recording.mp4"])` — Android screenrecord runs until killed or hits 3-min limit.
    - `stopRecording`: Send SIGINT to adb process, wait for exit, then `execFileAsync("adb", ["-s", deviceId, "pull", "/sdcard/simvyn_recording.mp4", outputPath])` to pull the file, then cleanup the device file.

    Note: The Android stopRecording needs the deviceId to pull the file. Adjust the interface to pass `deviceId` and `outputPath` to stopRecording: `stopRecording?(process: ChildProcess, deviceId: string, outputPath: string): Promise<void>`. The iOS implementation ignores the extra params since the file is already at outputPath.
  </action>

  <action>
    Create `packages/modules/screenshot/` following the same patterns as log-viewer and app-management modules.

    **package.json**: Same structure as log-viewer — name `@simvyn/module-screenshot`, type module, dependencies on `@simvyn/types`, `@simvyn/core`, `@simvyn/server`.

    **tsconfig.json**: Same as log-viewer — extends root, composite true, NodeNext.

    **recorder.ts** — Screen recording lifecycle manager:
    - Maintains a Map<string, { process: ChildProcess, outputPath: string, deviceId: string, startTime: number }> keyed by deviceId (one recording per device).
    - `startRecording(deviceId: string, platform: Platform, adapter: PlatformAdapter, outputDir: string): { outputPath: string }` — generates filename with timestamp (`recording-{deviceId}-{timestamp}.mp4`), calls adapter.startRecording, stores in map. Throws if already recording this device.
    - `stopRecording(deviceId: string, adapter: PlatformAdapter): Promise<{ outputPath: string, duration: number }>` — calls adapter.stopRecording with stored process/deviceId/outputPath, removes from map, returns metadata.
    - `isRecording(deviceId: string): boolean`
    - `getActiveRecordings(): string[]` — returns device IDs with active recordings.

    **routes.ts** — Fastify routes (prefix will be `/api/modules/screenshot`):
    - `POST /capture/:deviceId` — Capture screenshot. Create output dir `~/.simvyn/screenshot/captures/` if needed. Generate filename `screenshot-{deviceId}-{Date.now()}.png`. Call adapter.screenshot(). Save metadata to history JSON. Return `{ filename, path, timestamp, deviceId }`.
    - `POST /record/start/:deviceId` — Start recording. Use recorder.startRecording(). Broadcast recording state via WS. Return `{ recording: true, deviceId }`.
    - `POST /record/stop/:deviceId` — Stop recording. Use recorder.stopRecording(). Save metadata to history. Broadcast state via WS. Return `{ filename, path, duration, deviceId }`.
    - `GET /history` — List all screenshots and recordings from history JSON file. Return array of `{ type: "screenshot"|"recording", filename, path, timestamp, deviceId, deviceName, duration? }`.
    - `GET /download/:filename` — Send file from captures/recordings directory. Use `reply.sendFile` or manual stream with Content-Disposition header.
    - `GET /recording-status` — Return map of currently recording device IDs.

    Use `createModuleStorage("screenshot")` from `@simvyn/core` for history persistence (key: "history", stores array of capture metadata).

    **ws-handler.ts** — WS namespace "screenshot":
    - On subscribe: send current recording status for all devices.
    - Broadcast `recording-started` and `recording-stopped` events when recording state changes.
    - Pattern: same as log-viewer ws-handler — register on WsBroker with channel "screenshot".

    **manifest.ts** — Module manifest:
    - name: "screenshot", icon: "camera", capabilities: ["screenshot", "screenRecord"]
    - register: register routes + WS handler
    - cli: Two commands:
      1. `simvyn screenshot <device> [--output <path>]` — capture screenshot. If no --output, save to cwd with generated name. Print path to stdout.
      2. `simvyn record <device> [--output <path>]` — start recording. Print "Recording... press Ctrl+C to stop". On SIGINT, stop recording, pull file, print path. Same CLI adapter pattern as log-viewer (create own adapters + DeviceManager + ProcessManager).

    Follow established patterns:
    - Device lookup by ID or prefix match
    - Guard against `avd:` prefix for Android
    - `type {} from "@simvyn/server"` for Fastify decorator types
    - Per-registrar try/catch in CLI module discovery
    - execFile (not exec) for all shell commands
  </action>
