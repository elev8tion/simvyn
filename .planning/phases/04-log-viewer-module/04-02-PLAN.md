**routes.ts** — Create `logRoutes` Fastify plugin (following app-management/routes.ts pattern):

One route: `GET /export/:deviceId` with query param `format` (default `"json"`, also accepts `"text"`).
- Look up the active streamer for the device (import a shared `getActiveStreamer` or accept it won't have history if no stream is active — return empty if no streamer)
- Actually, the WS handler manages streamers. The route needs access to the streamer map. Use Fastify's decorator pattern: the WS handler decorates `fastify` with `logStreamers` (a Map<string, { streamer, clients }>) so routes can read history.
- Alternatively, keep it simpler: the ws-handler exports a `getLogHistory(deviceId)` function that the route imports. The ws-handler module maintains the map internally and exposes a getter.
- For JSON format: return `{ deviceId, entries: LogEntry[], count: number }`
- For text format: set content-type `text/plain`, return each entry as `[timestamp] [LEVEL] processName: message` one per line
- If no history for device, return empty array/string (not 404)
- Import `type {} from "@simvyn/server"` for Fastify decorator types (same pattern as app-management)

**ws-handler.ts** — Create `registerLogWsHandler(fastify)` function:

Data structures:
```typescript
interface StreamRef {
  streamer: ReturnType<typeof createLogStreamer>;
  clients: Set<WebSocket>;
}
const activeStreams = new Map<string, StreamRef>();
```

Export `getLogHistory(deviceId)` function that returns `activeStreams.get(deviceId)?.streamer.history ?? []`.

Register channel `"logs"` with `wsBroker.registerChannel("logs", handler)`.

**Message types handled:**

1. `"start-stream"` — payload: `{ deviceId }`
   - Validate device exists via `deviceManager.devices.find()`, must be booted
   - If `activeStreams` has this deviceId: add socket to clients set, send `"stream-started"` back, return (reuse existing child process)
   - Else: create new LogStreamer via `createLogStreamer({ deviceId, platform: device.platform, processManager: fastify.processManager, onFlush })` where `onFlush` broadcasts `"log-batch"` with `{ deviceId, entries }` to all clients in the StreamRef's client set (NOT wsBroker.broadcast — only send to clients subscribed to THIS device's stream)
   - Use `wsBroker.send()` to each client in the set individually
   - Store in activeStreams map, start the streamer, send `"stream-started"` to requesting socket

2. `"stop-stream"` — payload: `{ deviceId }`
   - Remove socket from StreamRef's client set
   - If client set becomes empty: stop the streamer, delete from activeStreams map
   - Send `"stream-stopped"` to requesting socket

**Client disconnect handling:**
- Listen for socket `"close"` event. On close, iterate all activeStreams and remove this socket from any client sets. If any client set becomes empty, stop that streamer and delete from map.
- The WsBroker's `socket.on("close")` in ws-broker.ts only removes from `clients` set. The channel handler needs its own cleanup. Since the channel handler receives the socket on each message, track connected sockets separately.
- Implementation: maintain a `WeakMap<WebSocket, Set<string>>` mapping each socket to the deviceIds it's streaming. On `"start-stream"` add the deviceId. On `"stop-stream"` remove it. 
- Register a socket close listener when first seeing a socket: `socket.on("close", () => cleanupSocket(socket))`. Use a WeakSet to track which sockets already have the listener attached.

**onFlush callback:**
- When LogStreamer flushes a batch, iterate the StreamRef's clients set
- For each client with `readyState === OPEN`, use `wsBroker.send(client, "logs", "log-batch", { deviceId, entries })`
- This ensures only clients subscribed to that specific device's logs receive the batch

import type { SimvynModule } from "@simvyn/types";

  name: "logs",
  version: "0.1.0",

  icon: "scroll-text",

  async register(fastify, _opts) {
    await fastify.register(logRoutes);
    registerLogWsHandler(fastify);
  },

  capabilities: ["logs"],
};
