  <action>
1. Install react-router v7: `npm install react-router -w @simvyn/dashboard`

2. Update `App.tsx`:
   - Import `BrowserRouter`, `Routes`, `Route`, `Navigate` from `react-router`
   - Wrap the entire app in `<BrowserRouter>` (outside `<WsProvider>`)
   - Define routes:
     - `/:moduleName` ‚Üí renders `<AppContent />` (the existing layout)
     - `/` ‚Üí renders `<AppContent />` (no module selected = home/empty state)
     - `*` ‚Üí `<Navigate to="/" replace />` (catch-all redirect)
   - Inside `AppContent`, add a `<RouterSync />` component (defined in same file or inline) that:
     - Reads `useParams().moduleName` from react-router
     - On mount and when `moduleName` changes, calls `setActiveModule(moduleName)` if the module exists in the modules list, otherwise navigates to `/`
     - This is the URL‚Üístore sync direction

3. Update `module-store.ts`:
   - No structural changes needed ‚Äî `setActiveModule` stays as-is
   - The store remains the source of truth for which module is active; the URL just drives it on load/navigation

4. Update `Sidebar.tsx`:
   - Import `useNavigate` from `react-router`
   - Replace `onClick={() => setActiveModule(mod.name)}` with `onClick={() => navigate(`/${mod.name}`)}` 
   - This ensures clicking a sidebar icon changes the URL, which then syncs to the store via RouterSync
   - Keep `activeModule` from store for highlighting (not from URL) ‚Äî store is always in sync

Key design decisions:
- URL is the entry point, store is the runtime state. RouterSync bridges them unidirectionally (URL‚Üístore). Sidebar navigates via URL (store updates as consequence).
- Module slugs in URL match module `name` field exactly (e.g., `/logs`, `/location`, `/deep-links`, `/crash-logs`)
- No nested routes needed ‚Äî all modules are flat, single-level
- Server already has SPA fallback (`setNotFoundHandler` ‚Üí `sendFile("index.html")`) so direct URL access works without server changes
