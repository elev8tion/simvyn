**Update `packages/modules/collections/routes.ts`:**

Import `getStarterCollections` and `STARTER_IDS` from `./starter-collections.js`.

Modify the `readCollections` helper: After reading from storage, if the result is null or empty (first launch), seed the starter collections:
```typescript
async function readCollections(): Promise<Collection[]> {
  const stored = await storage.read<Collection[]>("collections");
  if (stored && stored.length > 0) return stored;
  
  const starters = getStarterCollections();
  await writeCollections(starters);
  return starters;
}
```

This way:
- First launch: no collections on disk → seed starters → return them
- After user deletes all: they get starters back (this is acceptable — if they want empty they can delete starters)
- Existing users with collections: starters are never injected (they already have data)

Note: Do NOT change the CLI commands. `simvyn collections list` and `simvyn collections apply` already work correctly from Phase 24 — they read from the same storage file. The seeding happens when the server starts and the GET / route is first called, which means the CLI (which reads storage directly) won't auto-seed. This is fine — starters are a dashboard-first experience. If a user runs the CLI first, they'll see "No collections saved" which is expected.
