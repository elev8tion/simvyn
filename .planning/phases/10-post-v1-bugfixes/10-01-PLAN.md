<task type="auto">
  <name>Task 1: Fix z-index stacking for dock tooltips and device selector dropdown</name>
  <files>
    packages/dashboard/src/main.css
    packages/dashboard/src/components/TopBar.tsx
  </files>
  <action>
**Bug analysis:**
The layout is: `TopBar` (header) above a `flex` row of `Sidebar` + `ModuleShell`. ModuleShell has `relative flex-1 overflow-hidden` with absolutely-positioned children (`absolute inset-0`). This creates a stacking context that sits on top of:
1. The sidebar's tooltip overflow (tooltip positioned `left: 100%` extends into the content area)
2. The TopBar's dropdown overflow (dropdown positioned `top: 100%` extends below the header into the content area)

**Fix for dock tooltips (Bug 1):**
In `main.css`, add `z-index: 20` to `.dock-sidebar`. This establishes the sidebar's stacking context above the content area. The tooltip already has `z-index: 100` (relative to its parent), so it will render above content once the sidebar itself is above the content.

Also add `position: relative` to `.dock-sidebar` so the z-index takes effect (z-index only works on positioned elements).

**Fix for device selector dropdown (Bug 2):**
In `TopBar.tsx`, add `relative z-30` to the `<header>` className. The `relative` is needed for z-index to take effect. `z-30` (Tailwind: z-index 30) puts the header above both the sidebar (z-20) and the content area, so the dropdown (which already has `z-50` relative to the header) renders correctly.

Do NOT change ModuleShell — it doesn't need a lower z-index, the other elements just need to be explicitly higher.

  <name>Task 2: Fix location module map rendering (height chain)</name>
  <files>
    packages/dashboard/src/components/ModuleShell.tsx
  </files>
  <action>
**Bug analysis:**
The height chain from the app root to the Leaflet map is:

```
div.flex.h-screen.flex-col          ← full viewport ✓
  → div.flex.flex-1.overflow-hidden  ← fills remaining ✓
    → ModuleShell: div.relative.flex-1.overflow-hidden  ← flex-1 ✓
      → div.absolute.inset-0.overflow-auto  ← fills ModuleShell ✓
        → motion.div (AnimatedPanel)  ← ❌ NO HEIGHT SET
          → LocationPanel: div.location-panel (height: 100%)  ← 100% of... nothing
            → div.location-map (flex: 1)  ← collapses to 0
              → MapContainer (height: 100%)  ← 0px
```

The `motion.div` from `AnimatedPanel` breaks the height chain. It has no height constraint, so its child's `height: 100%` resolves to 0.

**Fix:**
In `ModuleShell.tsx`, change the `AnimatedPanel` wrapper's `motion.div` to include `className="h-full"`. This makes the motion.div fill its absolute-positioned parent, and the location panel's `height: 100%` resolves correctly.

Also ensure the outer `div.absolute.inset-0` doesn't use `overflow-auto` which can break flex height calculations for the location panel. Change to `overflow-hidden` — individual panels handle their own scrolling.

Wait — actually `overflow-auto` is needed for other panels (logs, settings) that scroll. The real fix is simpler: just add `h-full` to the `motion.div` AND to the wrapping `Suspense` or ensure the chain is unbroken.

The precise fix:
1. In `AnimatedPanel`, change `<motion.div animate={controls}>` to `<motion.div className="h-full" animate={controls}>`
2. This ensures the motion.div fills the absolute-positioned parent, and LocationPanel's `height: 100%` resolves correctly
3. Other panels that don't use `height: 100%` (they use padding/flex internally) are unaffected since `h-full` just means "be as tall as parent" — content-based panels will still work fine inside a container that's viewport-height
