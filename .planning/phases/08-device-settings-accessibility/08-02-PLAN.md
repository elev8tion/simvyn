  <action>
Create the settings panel as a single-page layout with sections (NOT tabs — settings are toggles/controls that benefit from seeing everything at once). Use the same glass-panel aesthetic and DeviceSelector pattern as other panels.

1. **SettingsPanel.tsx** — Main panel component:
   - DeviceSelector at top (same pattern as PushPanel/DeepLinksPanel)
   - On device select, fetch capabilities from `GET /api/modules/settings/capabilities?deviceId=<id>` to know which sections to show
   - Sections rendered in order: Appearance, Status Bar (iOS), Permissions, Locale, Accessibility
   - Each section in a glass-panel card with a header label
   - Show/hide sections based on capabilities response (e.g., hide Status Bar section for Android devices)

   **Appearance section** (inline in SettingsPanel, simple enough):
   - Two toggle buttons side by side: "Light" and "Dark" with sun/moon icons (use Lucide Sun and Moon)
   - On click, POST to `/api/modules/settings/appearance` with `{ deviceId, mode }`
   - Show toast on success ("Dark mode enabled")
   - Glass-panel segment control style (similar to database tab bar)

   **Locale section** (inline in SettingsPanel):
   - Text input for locale code (e.g., "en_US", "ja_JP", "fr_FR")
   - "Apply" button next to input
   - POST to `/api/modules/settings/locale`
   - Toast: "Locale set to {locale} (reboot may be required)"

2. **StatusBarSection.tsx** — iOS status bar overrides:
   - Form fields for: Time (text input), Battery Level (range slider 0-100), Battery State (dropdown: charged/charging/discharging), Cellular Bars (dropdown 0-4), WiFi Bars (dropdown 0-3), Operator Name (text input), Data Network (dropdown: wifi/3g/4g/lte/lte-a/lte+/5g/5g+/5g-uwb)
   - "Apply Overrides" button → POST to `/api/modules/settings/status-bar` with only non-empty fields in overrides
   - "Clear" button → POST to `/api/modules/settings/status-bar/clear`
   - Compact 2-column grid layout for form fields
   - Show iOS-only badge (same pattern as push panel)

3. **PermissionsSection.tsx** — Permission management:
   - Requires app selection — reuse the same app-selector dropdown pattern (fetch apps list from `/api/modules/apps/list` endpoint)
   - Dropdown for permission selection. iOS permissions: photos, camera, microphone, contacts, calendar, reminders, location, motion, health, homekit, siri, speech, media-library, bluetooth-peripheral. Android permissions: CAMERA, READ_CONTACTS, RECORD_AUDIO, ACCESS_FINE_LOCATION, READ_CALENDAR, READ_EXTERNAL_STORAGE, SEND_SMS, CALL_PHONE.
   - Three action buttons: "Grant" (green), "Revoke" (red), "Reset All" (orange, iOS-only)
   - POST to respective endpoints with deviceId, bundleId, permission
   - Toast on success

4. **AccessibilitySection.tsx** — Accessibility quick presets:
   - Content Size: dropdown with all valid sizes (extra-small through accessibility-extra-extra-extra-large). Show only when `contentSize` capability is true (iOS).
   - Increase Contrast: toggle switch. Show only when `increaseContrast` capability is true (iOS).
   - TalkBack: toggle switch. Show only when `talkBack` capability is true (Android).
   - "Accessibility Presets" quick buttons at top:
     - "Large Text" → sets content-size to extra-large
     - "Extra Large" → sets content-size to accessibility-large
     - "High Contrast" → enables increase contrast
     - "Default" → resets content-size to medium, disables increase contrast
   - Each preset button styled as a small glass pill
   - Only show presets relevant to the platform (iOS presets for iOS, TalkBack toggle for Android)

All components follow established patterns:
- Use `fetch()` for API calls (no axios)
- Use `sonner` toast for success/error feedback
- Use oklch glass-panel styling with Tailwind classes
- Consistent error handling: try/catch, toast.error on failure
