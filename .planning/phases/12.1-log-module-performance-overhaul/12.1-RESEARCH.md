# Phase 12.1: Log Module Performance Overhaul - Research

**Researched:** 2026-02-26
**Domain:** React virtualized lists, WebSocket pagination, device log management
**Confidence:** HIGH

## Summary

This phase overhauls the log module from a "dump everything into memory" model to a production-grade paginated, virtualized, reverse-chronological log viewer. The core changes span three layers: (1) server-side history buffer pagination via new WS message types, (2) client-side virtual list rendering with react-virtuoso, and (3) new device log clearing commands.

**react-virtuoso** is the clear choice over `@tanstack/react-virtual` for this use case. Virtuoso handles variable-height items automatically (no manual measurement), has built-in `firstItemIndex` + `startReached` for reverse infinite scrolling (prepending older items), and provides `endReached` + `followOutput` for auto-scrolling to new items. TanStack Virtual is headless and requires significant manual work for dynamic heights and scroll position maintenance during prepend — exactly the hard parts of this feature.

**Primary recommendation:** Use react-virtuoso v4 with `firstItemIndex` for reverse-chronological pagination, add `get-history` / `clear-device-logs` WS message types to the existing logs channel, and change ModuleShell to conditionally render (not display:none) the logs panel.

### Pattern 1: Reverse-Chronological Virtualized List with react-virtuoso
**What:** Display newest logs at top, load older logs when user scrolls down, auto-scroll to new entries when at top.
**When to use:** Any reverse-chronological feed with pagination.
**Example:**
```typescript
// Source: https://virtuoso.dev/react-virtuoso/api-reference/virtuoso/
import { Virtuoso, type VirtuosoHandle } from 'react-virtuoso'

// Data model: entries stored newest-first (index 0 = newest)
// firstItemIndex tracks the virtual starting index for prepending older items

const INITIAL_INDEX = 100_000 // large number to allow prepending
const PAGE_SIZE = 500

function LogList() {
  const virtuosoRef = useRef<VirtuosoHandle>(null)
  const [firstItemIndex, setFirstItemIndex] = useState(INITIAL_INDEX)
  const entries = useLogStore(s => s.entries) // newest-first array

  const loadOlderLogs = useCallback(() => {
    // Request older batch from server via WS
    // On response, prepend to entries and decrease firstItemIndex
    requestOlderLogs().then(olderBatch => {
      setFirstItemIndex(prev => prev - olderBatch.length)
      prependEntries(olderBatch)
    })
  }, [])

  return (
    <Virtuoso
      ref={virtuosoRef}
      firstItemIndex={firstItemIndex}
      initialTopMostItemIndex={0} // start at newest (top)
      data={entries}
      startReached={loadOlderLogs} // fires when scrolled to top (oldest visible)
      followOutput="smooth" // auto-scroll when new entries arrive at index 0
      itemContent={(index, entry) => <LogRow entry={entry} />}
      increaseViewportBy={200}
    />
  )
}
```

### Pattern 2: Server-Side History Pagination via WS
**What:** Extend the existing WS logs channel with a `get-history` request/response pattern.
**When to use:** Client needs a page of older logs from the server's in-memory buffer.
**Example:**
```typescript
// Server: ws-handler.ts — new message type in the existing registerChannel
if (type === "get-history") {
  const { deviceId, before, limit } = data as {
    deviceId: string; before?: number; limit?: number
  }
  const pageSize = Math.min(limit ?? 500, 500)
  const history = getLogHistory(deviceId) // chronological order
  // Slice from history: entries before the given cursor index
  const endIdx = before ?? history.length
  const startIdx = Math.max(0, endIdx - pageSize)
  const page = history.slice(startIdx, endIdx)

  wsBroker.send(socket, "logs", "history-page", {
    deviceId,
    entries: page.reverse(), // send newest-first
    cursor: startIdx,        // client uses this for next page
    hasMore: startIdx > 0,
  }, requestId)
}
```

### Pattern 3: Conditional Rendering for Unmount Cleanup
**What:** Replace display:none/block with conditional rendering for the logs panel so it fully unmounts.
**When to use:** Panels that hold large data (logs) should release memory when hidden.
**Example:**
```typescript
// ModuleShell.tsx — special case for heavy panels
const UNMOUNT_WHEN_HIDDEN = new Set(["logs"])

{moduleList.map((mod) => {
  const Panel = registry.getPanel(mod.name)
  const isActive = activeModule === mod.name
  const shouldUnmount = UNMOUNT_WHEN_HIDDEN.has(mod.name) && !isActive

    <div
      key={mod.name}
      className="absolute inset-0 overflow-auto"
      style={{ display: isActive ? "block" : "none" }}
    >
      {Panel && (
        <Suspense fallback={<LoadingSkeleton />}>
          <AnimatedPanel isActive={isActive}><Panel /></AnimatedPanel>
        </Suspense>
      )}
    </div>
  )
})}
```

### Pattern 4: Device Log Clearing
**What:** Clear device-side logs via platform command, then reset client state.
**When to use:** User wants fresh log output without accumulated noise.
**Example:**
```typescript
// Server: ws-handler.ts
if (type === "clear-device-logs") {
  const { deviceId } = data
  const device = deviceManager.devices.find((d: Device) => d.id === deviceId)

  if (device?.platform === "android") {
    // adb logcat -c clears the device log buffer
    processManager.spawn("adb", ["-s", deviceId, "logcat", "-c"])
  } else if (device?.platform === "ios") {
    // iOS has no equivalent command.
    // Restart the log stream to get a fresh start.
    const ref = activeStreams.get(deviceId)
    if (ref) {
      ref.streamer.stop()
      ref.streamer.start()
    }
  }

  // Clear the server-side history buffer
  const ref = activeStreams.get(deviceId)

  // ...
  wsBroker.send(socket, "logs", "device-cleared", { deviceId }, requestId)
```

### Anti-Patterns to Avoid
- **Storing all log entries in a single flat array with no cap in the virtualized model:** Even with virtualization, keeping 500K entries in JS memory causes GC pauses. Keep a reasonable cap (e.g. 10K loaded entries on client, rely on server buffer for history).
- **Using array index as key in Virtuoso:** Use a stable key derived from timestamp + pid + index to avoid flicker on prepend.
- **Filtering 50K entries on every render via selector:** The current `selectFilteredEntries` runs `.filter()` on every store change. With pagination, the dataset is smaller, but memoize the selector result properly.
- **CSS margins on Virtuoso items:** react-virtuoso's ResizeObserver measurement does not include CSS margins. Use padding instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Virtual list rendering | Custom windowing with absolute positioning | react-virtuoso `Virtuoso` | Variable heights, scroll position maintenance on prepend, auto-measurement — months of edge-case work |
| Reverse infinite scroll | Manual scroll position calculation on prepend | react-virtuoso `firstItemIndex` | Scroll jumps are the #1 bug in hand-rolled implementations |
| Dynamic row height measurement | ResizeObserver + height cache | react-virtuoso (built-in) | Handles async resize, content changes, font loading |

**Key insight:** The hardest part of this feature is maintaining scroll position when prepending older items. react-virtuoso's `firstItemIndex` mechanism solves this declaratively — you just decrease the number and it handles the rest.

## Common Pitfalls

### Pitfall 1: Scroll Position Jumps When Prepending Items
**What goes wrong:** When older logs are loaded and prepended to the array, the scroll position jumps to a different place because the browser inserted content above the current viewport.
**Why it happens:** Standard DOM behavior — inserting elements above the scroll position changes scrollTop.
**How to avoid:** Use react-virtuoso's `firstItemIndex` pattern. Set a high initial value (e.g. 100,000) and decrease it by the number of prepended items. Virtuoso compensates internally.
**Warning signs:** Users report "scroll jumping" when loading more logs.

### Pitfall 2: Memory Bloat from Unbounded Log Accumulation
**What goes wrong:** New logs stream in via WebSocket continuously. Without a cap, client memory grows indefinitely.
**Why it happens:** The current architecture pushes ALL logs to the client and caps at 50K — still too many for a paginated model.
**How to avoid:** Cap the client-side loaded entries at ~5,000-10,000. When new entries arrive and the cap is exceeded, trim the oldest loaded entries. The server still has the full history buffer (capped at 10K). Accept that extremely old logs may no longer be accessible.
**Warning signs:** Dashboard memory usage grows steadily over time.

### Pitfall 3: Race Conditions Between Streaming and History Loading
**What goes wrong:** While a `get-history` request is in-flight, new streaming logs arrive. The client may insert the history response at the wrong position or create duplicates.
**Why it happens:** Async WS messages can arrive in any order.
**How to avoid:** Use a loading flag/ref to prevent concurrent history requests. Deduplicate by checking timestamps/cursor positions. Use a monotonic cursor (buffer index) rather than timestamp-based cursors.
**Warning signs:** Duplicate log entries appearing, or gaps in the log timeline.

### Pitfall 4: Attempting `xcrun simctl` Log Clearing
**What goes wrong:** Developer assumes there's an iOS simulator equivalent of `adb logcat -c`.
**Why it happens:** There isn't one. The `log` command on macOS/iOS doesn't support clearing the log store from simctl.
**How to avoid:** For iOS, restart the log stream process to get a fresh view. The "clear" on iOS is effectively a UI-only + stream restart, not a device-side wipe.
**Warning signs:** Build failures or runtime errors from non-existent simctl subcommands.

### Pitfall 5: ModuleShell Unmount Breaking Other Panels
**What goes wrong:** Making conditional rendering the default for all panels breaks state persistence for panels that benefit from display:none.
**Why it happens:** Other panels (maps, settings) may want to persist state across module switches.
**How to avoid:** Only apply conditional unmount to explicitly listed "heavy" panels (logs). Keep display:none for all other panels.
**Warning signs:** Users lose state in other panels when switching back.

### react-virtuoso Reverse Infinite Scroll (Verified Pattern)
```typescript
// Source: https://virtuoso.dev/react-virtuoso/api-reference/virtuoso/
// Key props for reverse-chronological log viewing:

<Virtuoso
  // High starting index, decrease to prepend older items
  firstItemIndex={firstItemIndex}

  // Start showing the newest item (top of list)
  initialTopMostItemIndex={0}

  // Data array in newest-first order
  data={entries}

  // Called when user scrolls to oldest loaded item (top of virtual range)
  startReached={() => loadOlderPage()}

  // Called when user scrolls to newest loaded item (bottom of virtual range)
  // Can use for "load newer if we trimmed" or just ignored
  endReached={() => {}}

  // Auto-follow new items when user is at the newest position
  followOutput="smooth"

  // Render each log row
  itemContent={(index, entry) => <LogRow entry={entry} />}

  // Pre-render 200px above/below viewport for smoother scrolling
  increaseViewportBy={200}

  // Stable keys to prevent flicker on prepend
  computeItemKey={(index, entry) => `${entry.timestamp}-${entry.pid}-${index}`}
/>
```

### Zustand Store Redesign
```typescript
// Key changes to log-store.ts:
interface LogStore {
  entries: LogEntry[]           // newest-first order
  firstItemIndex: number        // for react-virtuoso, starts high (e.g. 100_000)
  cursor: number | null         // server-side cursor for next page
  hasMore: boolean              // whether more history is available
  isLoadingHistory: boolean     // prevent concurrent fetches
  // ... existing filter state unchanged
  prependHistory: (batch: LogEntry[], cursor: number, hasMore: boolean) => void
  addNewBatch: (batch: LogEntry[]) => void  // new logs arrive at beginning
  reset: () => void             // full cleanup on unmount
}
```

## Open Questions

1. **Server buffer size vs. client pagination**
   - What we know: Server caps at 10K entries, client at 50K. With pagination, client will hold fewer.
   - What's unclear: Should the server buffer increase to support deeper history scrollback? 10K at 500/page = only 20 pages.
   - Recommendation: Keep server at 10K for now. If users need deeper history, that's a separate feature (log persistence to disk).

2. **New log arrival position in reverse-chronological view**
   - What we know: New logs should appear at the top (index 0 in the data array).
   - What's unclear: Should we auto-scroll to top when new logs arrive, or only when user is already at top?
   - Recommendation: Use Virtuoso's `followOutput` — only auto-scroll if user is at the top (viewing newest). If they've scrolled down to older logs, don't interrupt.

3. **Search across loaded vs. all history**
   - What we know: Current search filters client-side on all loaded entries. With pagination, only loaded pages are searchable.
   - What's unclear: Is searching only loaded pages sufficient, or do users expect to search all server history?
   - Recommendation: Search loaded pages only for v1. This is simpler, faster, and covers the common case. Server-side search can be added later if needed.
