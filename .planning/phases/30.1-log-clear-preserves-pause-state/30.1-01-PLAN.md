<objective>
Fix the log store's `clear` action so it no longer resets the `isPaused` state to `false`.

Purpose: When a user pauses log streaming and then clicks "Clear" (UI-only clear) or "Purge Device" (device-level clear), the pause state should be preserved. Currently, `clear()` unconditionally sets `isPaused: false`, which triggers the pause/resume `useEffect` in LogPanel to restart the stream — unpausing logs without user intent. (GitHub issue #5)

Output: Fixed log-store.ts with isPaused preserved across clear operations.
</objective>

In `packages/dashboard/src/panels/logs/stores/log-store.ts`, modify the `clear` action (lines 76-83) to remove `isPaused: false` from the state update.

Before:
```ts
clear: () =>
    set({
        entries: [],
        firstItemIndex: INITIAL_INDEX,
        cursor: null,
        hasMore: true,
        isPaused: false,
    }),
```

After:
```ts
clear: () =>
    set({
        entries: [],
        firstItemIndex: INITIAL_INDEX,
        cursor: null,
        hasMore: true,
    }),
```

This is safe because:
- The `resume()` action already sets `isPaused: false` independently. The resume path in LogPanel's useEffect (line 91) calls `clear()` AFTER `isPaused` was already set to `false` by the `resume()` action, so the clear doesn't need to duplicate this.
- The `reset()` action (used on full module teardown) still resets `isPaused` via `...initialState`.
- The `addNewBatch` action already guards with `if (s.isPaused) return s`, so paused + cleared state won't accumulate phantom entries.

Do NOT modify LogPanel.tsx — the `handleDeviceCleared` callback correctly calls `clear()`, and now it will preserve the pause state as intended.
